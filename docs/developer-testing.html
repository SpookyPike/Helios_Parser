Use targeted runs for the area you touched before falling back to broad suite execution. When a change affects shell/viewer/derived routing, you usually need both a focused derived test and at least one shell-level test.

What to run for common changes

Parser, reader, schema

Run coordinate, run-status, reader, and smoke coverage. Use at least one real damaged/truncated or legacy-style file if the parser changed.

Viewer layout or settings

Run viewer phase tests, shell settings propagation, and manual screenshot validation for the touched workflow.

Derived orchestration or caching

Run backend optimization, derived stability, and WaveFront/Preheat UI tests. Check that legacy Shock is still fast and that advanced tabs remain lazy.

Plasmon/XRTS spectral work

Run the plasmon phase and UI suites, then use the validation scripts for benchmark overlays, cache reuse, snapshot refresh, and LOS-vs-effective-state checks before shipping docs or UI claims.

WaveFront or Preheat UI

Run test_wave6_derived.py, then do manual visual validation on at least one real run where the affected advanced tab has meaningful content.

Manual QA expectations

Automated tests are not enough for major UI or orchestration changes. Manual QA is expected when:

  • tabs, selectors, splitters, or scroll behavior changed
  • busy/progress behavior changed
  • WaveFront or Preheat UI changed
  • screenshots or docs are being updated
  • a fix is explicitly about interaction feel or stale UI state

Performance and timing checks

Use the existing instrumentation rather than guessing. Important checks include:

  • legacy Derived open remains fast
  • WaveFront first-load timing is visible and cached reopen is meaningfully faster
  • Preheat reuses WaveFront/interface context and does not force full recompute on ROI-only changes
  • snapshot browsing stays lightweight after advanced data are already loaded
  • heavy background work still shows progress/busy feedback
python scripts\bench_wave6_derived.py
python scripts\microbench_wave1.py
python scripts\bench_streaming_conversion.py
python scripts\bench_hdf5_access.py

Definition of done for nontrivial work

  1. The touched backend/UI path has focused automated regression coverage.
  2. Smoke coverage still passes.
  3. Manual QA was performed for interaction-heavy changes.
  4. Busy/progress and cache/lazy behavior still match the intended UX.
  5. Documentation was updated if the visible behavior or workflow changed.

For large features, also keep at least one real-file validation note. The most common late regressions are not numerical; they are stale state, lazy-load drift, broken tab ownership, and settings propagation mismatches.
