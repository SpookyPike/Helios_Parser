High-level layers

src/
  helios_parser/   log parsing, schema normalization, HDF5 writing, lazy reader
  helios/          shared cache, tasks, runtime context, services, units
  helios_viewer/   production viewer window, settings, plotting, controller
  helios_analysis/ derived workspace and derived controller
  helios_app/      top-level Parse / View shell and About / release surface

The important rule is still the same: do not bypass layer boundaries without a specific reason. Parser remains the canonical raw log ingestion path, the reader remains the canonical HDF5 access layer, and derived modules should consume normalized runtime/derived data rather than re-reading ad hoc files or inventing parallel coordinate semantics.

End-to-end data flow

  1. Parser converts HELIOS logs into stabilized HDF5 with explicit center/edge coordinates, run status, and preserved diagnostics.
  2. The reader exposes stable accessors such as static/dynamic center/edge coordinates, fields, run status, and VISAR-readiness metadata.
  3. Viewer builds a RunContext and publishes shared raw-data cache entries for reuse.
  4. Derived uses load_run_data() and the shared cache instead of reopening HDF5 unnecessarily.
  5. The derived controller computes result models off the GUI thread and applies only the latest valid result.
Keep explicit coordinate semantics intact all the way through the stack. Geometry helpers already exist; do not add a second parallel interpretation of HELIOS coordinates.

Caches are bounded and intentionally observable. Hits, misses, evictions, and clear counts are lightweight engineering signals, not a production telemetry system.

Derived analysis architecture

helios.services.derived.analysis is the orchestrator for derived result construction. It uses result models, shared selection/weighting helpers, and modular contracts rather than one-off per-tab logic.

  • Legacy Shock remains a fast compatibility path and user-facing quick-look entry point.
  • WaveFront is the advanced tracker path built on typed branch and interface-event results.
  • Preheat is a separate advanced module that consumes wave-tracking results rather than living inside Shock or the tracker itself.
  • XRD / Plasmon / Transmission / Spectroscopy use shared reusable time-series paths and lazy tab-triggered loading.

Heavy work stays off the GUI thread. The workspace is a presentation layer; scientific computation belongs in the services layer.

Transmission is no longer just a Thomson-only quick look. The current module supports explicit selected-mode computation, a cold-opacity XCOM seam, and a deterministic precomputed fallback-table path for environments where the native XCOM backend cannot execute.

Plasmon has also grown into a staged spectral stack. Quicklook remains available, but the module now carries explicit RPA, Mermin, static-LFC, and manual finite-temperature Lindhard-family paths, a spectral-mode split between lightweight exploratory runs and stricter benchmark runs, clustered LOS integration, CSV export, and grouped warning/reporting fields so the UI can show when a result is degenerate, non-collective, out of domain, or only partially executed.

WaveFront, interface events, and Preheat

WaveFront

  • multi-branch tracking
  • branch support classes and significance
  • primary compressive branch selection
  • direction-aware filtering
  • interface-event summaries

Preheat

  • separate advanced tab
  • region-of-interest override
  • shock-relative and manual local time modes
  • target-entry timing, onset markers, depth, and budgets
  • reuse of cached WaveFront/interface context when only ROI changes

The design rule is that advanced modules consume typed results and existing helpers. They should not re-implement independent parser or geometry logic.

UI state, settings, and configuration flow

  • Viewer settings live in helios_viewer/settings.py and are persisted through QSettings.
  • Those settings flow into the derived workspace through the shell/controller path rather than a second settings store.
  • Global snapshot state is owned by the shell and synchronized into viewer and derived, with care around lazy advanced tabs.
  • Module-specific controls now belong to their module tabs rather than a misleading global control column.

If you change a setting that affects both viewer and derived behavior, audit the shell propagation path rather than patching one side only.

Transmission and XCOM seam

The cold-material path is intentionally isolated behind helios.services.derived.xcom_hook. That seam now handles:

  • material identity resolution from parsed EOS / opacity metadata
  • canonical machine keys for cold-material lookup
  • live backend health probing and loading
  • persistent cache lookup keyed by backend fingerprint and transport inputs
  • precomputed fallback-table lookup when the native backend cannot run

The important contract is that parsed EOS / opacity metadata is the practical material-identity source in this codebase. The machine path uses canonical keys such as cu, ch, c2h4o, and sio2, while the UI keeps the human-readable material display label.

The fallback table stores mu/rho in cm^2/g. HELIOS converts that to per-zone optical depth using tau = (mu/rho) * density_g_cm3 * path_length_cm, so the table path fills the same XCOM tau arrays and region budgets as the live backend path instead of bypassing downstream accounting.

Reasoning about performance-sensitive paths

  • Do not trigger advanced WaveFront or Preheat work during a normal legacy Shock refresh.
  • Do not put derived computation onto the GUI thread.
  • Do not recompute full advanced results just because the user moved the snapshot slider.
  • Prefer cached arrays and typed result reuse over rebuilding HDF5-backed objects.
  • Keep plot updates local to already-computed results whenever possible.
  • Do not re-probe XCOM backend health or re-open the fallback table on every trivial UI repaint; both are session-cached.

Instrumentation already exists. Use it. Any feature that changes derived orchestration should be checked against timing counters, cache reuse, and busy/progress behavior.

Adding new derived modules safely

  1. Add typed result models first.
  2. Use the existing orchestration and DerivedModuleContract path instead of introducing a second controller stack.
  3. Use the shared selection, weighting, geometry, and cache helpers before writing a new per-module reduction path.
  4. Keep heavy traces lazy and tab-triggered if the module is expensive.
  5. Add regression coverage for caching, tab activation, and UI surfacing, not only backend math.
