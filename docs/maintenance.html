Dependency management

The repository now publishes release-facing metadata through pyproject.toml. For the desktop application, the practical dependency set is the desktop extra: 

pip install -e .[desktop]

That covers the shipping desktop workflow:

  • numpy
  • h5py
  • PySide6
  • pyqtgraph
  • matplotlib for validation/export-supporting paths

Launching and packaging

Direct developer launch:

python -m helios_app

The shareable release bundle created in this pass includes top-level Windows and Linux launcher/bootstrap scripts. Those scripts create a local virtual environment, install the desktop extra, and launch the app from the unpacked bundle.

Maintenance rule: if the packaging or launch path changes, update both the scripts and the quick-start documentation together.

Documentation upkeep

  1. Update the relevant user/developer/testing/maintenance docs whenever visible behavior, workflow, or architecture changes.
  2. Refresh screenshots when the UI changes materially.
  3. Keep release number, date, and About dialog metadata in sync.
  4. Prefer a few clear screenshots to a giant stale archive.

Current screenshot maintenance commands:

python scripts\capture_docs_assets.py
python scripts\visual_qa_wavefront.py

Store documentation screenshots under docs/assets/screenshots/. Visual-QA artifacts can stay under outputs/screenshots/visual_qa/ and be copied into docs only when they represent the current shipping UI.

Sample and demo files

Do not blindly ship every large or historically convenient dataset. The release bundle should only contain small or moderate redistributable examples that are actually useful for onboarding.

Current bundle strategy:

  • include a small planar run for basic open/browse/derived checks
  • include a cylindrical run so coordinate semantics can be demonstrated
  • include one moderate quick-look derived example
  • do not bundle the heaviest layered advanced-analysis datasets unless there is a clear redistribution reason

Release and versioning expectations

The chosen release for this documentation and bundle pass is 0.9.0. The project is substantial and shareable, but still clearly pre-1.0 because the scientific stack continues to evolve and several advanced interpretations remain intentionally quick-look.

Maintenance checklist for a release-facing update:

  • update release metadata in About text, README, docs, and package metadata
  • refresh current screenshots
  • create and inspect the shareable zip bundle
  • verify top-level quick-start instructions against actual included scripts
  • check that docs links are valid inside the bundle tree

Common regression risks

  • viewer/derived settings drift, especially units and default coordinate handling
  • stale lazy-load cache invalidation between Shock, WaveFront, and Preheat
  • tab-local controls accidentally affecting the wrong module
  • snapshot navigation drifting out of sync between shell, viewer, and derived
  • breaking scroll, popup, or splitter behavior in dense scientific UIs
  • WaveFront or Preheat becoming eager by accident after a controller change
  • docs/screenshots quietly describing an older UI
If an architectural change touches parser, runtime, viewer, and derived together, plan a validation pass across all layers. These are the changes most likely to look locally correct while breaking real workflows.
